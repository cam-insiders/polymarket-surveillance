# Polymarket Insiders

Research code for detecting suspicious Polymarket trading behavior, backtesting
detectors on historical data, and comparing the system against baseline methods.

## Data Attribution

Historical Polymarket market and trade data for this project comes from
WarProxx's open-source [`warproxxx/poly_data`](https://github.com/warproxxx/poly_data)
pipeline. This repo was built around WarProxx's v1 historical data format: the
normalized `markets.csv` and `processed/trades.csv` files produced by that
pipeline.

This repo does not replace WarProxx's data pipeline. It consumes the files
produced by `poly_data`, converts the large trade CSV into `trades.db` for faster
local backtests, and runs detector/experiment code against that SQLite database.
Newer `poly_data` outputs or schema versions may need small loader/conversion
changes before they are compatible with these experiments.

## Setup

Use Python 3.11+ in a virtual environment.

```bash
python3 -m venv .venv
source .venv/bin/activate
python -m pip install --upgrade pip
pip install -r requirements.txt
```

Optional environment variables can be placed in `.env`:

```bash
POLYGONSCAN_API_KEY=...
OPENAI_API_KEY=...
```

`POLYGONSCAN_API_KEY` is only needed for Layer 2 USDC attribution when the local
USDC cache is missing entries. `OPENAI_API_KEY` is only needed for scripts that
classify markets with an LLM.

## Get Historical Data

This project expects historical data generated by WarProxx's
[`warproxxx/poly_data`](https://github.com/warproxxx/poly_data) repo, specifically
the v1-style CSV layout this code was originally written against. Clone or use
that repo first, then follow its README to produce the historical market and
trade CSVs. The important outputs from WarProxx's v1 data pipeline are:

```text
data/
  markets.csv
  processed/
    trades.csv
```

Then copy or symlink those files into this repo:

```bash
mkdir -p data/processed
cp /path/to/poly_data/data/markets.csv data/markets.csv
cp /path/to/poly_data/data/processed/trades.csv data/processed/trades.csv
```

Convert the trade CSV to SQLite before running serious experiments:

```bash
python convert_to_sqlite.py
```

This writes `data/trades.db`. The first conversion can take a while on the full
historical dataset, but experiments are much faster afterward.

If you have a self-contained dataset directory with `markets.csv` and `trades.csv`
at the same level, build its SQLite database with:

```bash
python scripts/csv_dir_to_sqlite.py --data-dir data/curated_fromvm
```

Then pass that directory to experiments with `--data-dir data/curated_fromvm`.

## Optional Precomputation

Precompute inferred market resolutions so repeated timeframe experiments do not
rescan all trades:

```bash
python -m experiments.precompute_inferred_resolutions \
  --data-dir data \
  --output-db inferred_resolutions.db
```

For Layer 2 attribution, prefetch Polygon USDC transfers for wallets in the trade
database:

```bash
python scripts/prefetch_usdc_transfers.py \
  --trades-db data/trades.db \
  --output-db data/usdc_transfers.db \
  --resume
```

Without a complete `data/usdc_transfers.db`, Layer 2 runs may query Polygonscan
at runtime if `POLYGONSCAN_API_KEY` is set.

## Run Experiments

Run commands from the repository root with the virtual environment activated.

### Timeframe Trade-Window Train/Backtest

This is the main experiment path. It selects markets that resolve inside the
train/test windows and replays only trades inside each corresponding window.

```bash
python -m experiments.timeframe_trade_window_train_backtest \
  --train-start 2025-02-01 \
  --train-end 2025-02-14 \
  --test-start 2025-02-15 \
  --test-end 2025-02-28 \
  --optimizer-mode coordinate_descent \
  --objective f0_5 \
  --n-starts 3 \
  --n-passes 2 \
  --enable-trade-prefilter \
  --min-usd-amount 500 \
  --enable-layer2-attribution \
  --enable-jump-anticipation \
  --enable-ja-optimization
```

Outputs are written under:

```text
experiments/results/timeframe_trade_window_train_backtest/
```

The run prints the selected best config path, typically named
`timeframe_best_config_*.json` or `timeframe_best_config_multi_start_*.json`.

To evaluate an existing config without re-optimizing:

```bash
python -m experiments.timeframe_trade_window_train_backtest \
  --config-path experiments/results/.../timeframe_best_config_....json \
  --test-start 2025-03-01 \
  --test-end 2025-03-15
```

### Full-Market Timeframe Train/Backtest

This variant selects markets by resolution window, but replays each selected
market's full trade history.

```bash
python -m experiments.timeframe_full_market_train_backtest \
  --train-start 2025-02-01 \
  --train-end 2025-02-14 \
  --test-start 2025-02-15 \
  --test-end 2025-02-28 \
  --optimizer-mode coordinate_descent \
  --objective f0_5 \
  --n-starts 3 \
  --n-passes 2
```

### Monthly Batch Runs

Run the built-in Jan-Apr 2025 monthly trade-window schedule:

```bash
python -m experiments.run_timeframe_f05_monthly_parallel
```

For trade-level objectives:

```bash
python -m experiments.run_timeframe_trade_monthly_parallel --objective cohens_d
```

Add `--sequential` if you do not want all months to run in parallel, or `--dry-run`
to print the child commands.

### SOTA Comparison

Compare the full system against baseline algorithms on explicit train/test
windows:

```bash
python -m experiments.compare_sota_timeframe \
  --train-start 2025-02-01 \
  --train-end 2025-02-14 \
  --test-start 2025-02-15 \
  --test-end 2025-02-28 \
  --optimizer-mode coordinate_descent \
  --n-passes 2 \
  --skip-random-baseline
```

### Category Evaluation

Optimize on a timeframe and evaluate by market category:

```bash
python -m experiments.timeframe_optimize_category_eval \
  --train-start-date 2025-02-01 \
  --train-end-date 2025-02-14 \
  --test-start-date 2025-02-15 \
  --test-end-date 2025-02-28 \
  --optimizer-mode coordinate_descent
```

Generate charts from a category-eval run:

```bash
python -m experiments.charting.category_eval_charts --list
python -m experiments.charting.category_eval_charts --run <run-folder-name>
```

Run the test suite:

```bash
pytest
```

```bash
python - <<'PY'
from backtesting.data_loader import HistoricalDataLoader

loader = HistoricalDataLoader(data_dir="data")
loader.load_data()
print(f"markets: {len(loader.markets_df):,}")
print(f"sqlite:  {loader.use_sqlite} ({loader.db_path})")
loader.close()
PY
```

## Live Detector

The live detector uses the current market slugs and thresholds in `config.py`:

```bash
python main.py
```

It writes local runtime databases such as `insider_trading_v1.db`,
`wallet_cache_v1.db`, and `clustering_v1.db`.
